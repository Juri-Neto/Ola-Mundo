Git - software de controle de versão (VCS) 

versionamento de codigo/software

Ajuda a controlar o versionamento de cada mudança feita em um codigo ou software, ajuda quando tem muitas pessoas trabalhando junto ou quando faz muitas modificações e adições no codigo. 

Antigamente utilizavam mais o modelo centralizado/linear que depende de uma conexão constante com um servidor que serve como repositório central. Salvando os commits diretamente.


Atualmente existe o modelo distribuido que permite que uma equipe trabalhe no mesmo codigo salvando seus commits em um repositório local (que é o Git) na própria maquina para só então fazer um push para o repositório remoto (que é o GitHub).

Vantagem de utilizar VCS: Controle de histórico, Trabalho em equipe, Ramificação do projeto, Segurança, Organização

issues - para solicitar ajuda para um problema.

Antes de abrir uma Issue, procure nas issues do repositório se uma Issue com o seu mesmo problema já foi aberto ou resolvido anteriormente.

Existem diversas opções de configuração de texto quando for abrir uma Issue ou adicionar um comentário. 

Em "Settings", "saved replies",  tem como adicionar respostas salvas previamente, como assinaturas de agradecimento e compartilhamento do seu repositório e perfil pessoal. 


pull request - encontrei um problema, analisei o codigo, aprendi como resolver, mexi no codigo, encontrei uma solução e solicitei um pull request para o dono analisar o codigo e só então aceitar ou não a modificação para o repositorio principal. 

como fazer isso: fazer um "fork", clonar o repositorio, criar uma branch para o problema especifico, realizar a alteração e commits para só então quando solucionado solicitar um pull request.

.md = linguagem Markdown - como o arquivo README.md

Markdown foi criado para ser uma linguagem de facil leitura, principalmente na leitura do codigo dele. Diferente do HTML que é uma linguagem que pode ser complexa a leitura do codigo.

Markdown é usado principalmente nos README.md, Issues e Pull Requests.

Pin issue - para selecionar e deixar em principal as Issues importantes que você queira destacar.

Lock conversation - limitar comentários de usuários que não são donos do repositório.

Quote reply - para responder comentários em Issues.

Branch - é um ramo do projeto.

Branch master/main é o ramo principal, produto final, versão corrente do software.

Podem existir outras branches como uma versão a frente da principal com novas features. Estas branches uma hora serão implementadas na principal. 

Branches podem ser criadas para resolver problemas especificos, cria-se a branch para solucionar o problema e depois junta para a principal.

Então quando criamos um repositório novo, é criado a branch master/main. Quando fazemos um commit, versionamos e criamos uma sub-versão desse software. Este commit fica salvo localmente, mas pode-se fazer um push para enviar do repositório local (git) para o repositório remoto (github) e guardar remotamente o software. 

A cada commit, um versionamento novo do software é criado e a cada push você salva ele remotamente no github. 

Evite fazer todos os commits na branch master/main, para exercícios simples não tem problema.

Commits errados na branch master podem comprometer o projeto completamente. O ideal é para cada feature nova, criar uma branch específica para desenvolver sem alterar a master/main. Isso se aplica a push também.
Fazendo os commits dessa forma, não altera o "tronco" do projeto que é a branch main. 
Tendo finalizado a nova feature ou correção de algum problema, é possível fazer um merge para o branch main do projeto.

É possível criar diversas branches, caso não consiga desenvolver na branch, é possível apagar a branch ou fazer um merge.

Na branch "main" criamos a estrutura principal do projeto, quando adicionar features e solucionar problemas criar as outras branches para manter estável a branch principal do software.
